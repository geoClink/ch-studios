The Problem

Small bakeries take orders by phone or Instagram DM, lose track of them, and pay 15–30% commission to DoorDash or Uber Eats to get discovered online. There's no system, no paper trail, and no way to know which items are low on stock before the morning rush. The goal was to build them a direct ordering system they own — one that replaces $200–500/month in tools like Square Online or Toast, costs them zero commission per order, and gives the owner a real dashboard instead of a phone full of DMs.

Overview

A production-capable full-stack ordering platform built to solve a real problem: small food businesses paying commissions to platforms they don't control and managing orders through DMs and phone calls. The Bakery Co. replaces that with a direct ordering system the business owns — real Stripe payments, a JWT-authenticated admin dashboard, transactional email and SMS, and three scheduled background jobs, all deployed on Render.

Two sides to the product: a customer-facing storefront and a full business management backend — order queue, menu CMS, revenue dashboard, one-click refunds, inventory management, and CSV export, all behind JWT authentication.

The architecture is intentionally forkable — swap the menu items, colors, and business name and the full ordering and admin platform is ready to redeploy for any food business.

Technical Decisions

Why Supabase

Supabase bundles Postgres, Auth, and file Storage in one service — no separate auth server, no third-party image host, and one set of credentials to manage. For a product this scope, that consolidation meaningfully reduces operational complexity.

Payment-First Flow

The Stripe charge is confirmed via webhook before the order is written to the database. This prevents unpaid orders from entering the system — if the charge fails or the server crashes between payment and save, no phantom order appears in the admin queue.

node-cron Inside the Server

Scheduled jobs run inside the Express process via node-cron. The tradeoff is simplicity vs. reliability — a server restart silently kills all scheduled jobs. A production deployment would use BullMQ or an external cron service.

Rate Limiting on Checkout

Rate limiting is applied specifically to the checkout endpoint, not just globally. Repeatedly hitting checkout could trigger Stripe fraud flags and put the merchant account at risk — that endpoint warranted its own stricter limit.

API-Level Auth vs. UI-Level Auth

The admin dashboard checks login state in the UI — but every API route it calls also verifies the JWT server-side. Hiding a button in the browser isn't security. Any HTTP client can call /api/admin/refund directly.

Forkable Architecture

The codebase is structured so the whole platform can be reskinned for any food business — different menu items, colors, and business name — in a few hours. The business logic is the same regardless of what's being sold.

Key Features

Stripe Checkout

Real payment processing via Stripe Elements — tip selection, delivery fee calculation, and minimum order enforcement all handled before the card is charged. Webhooks update order status in Supabase after payment confirmation.

Admin Dashboard

Full order management, revenue tracking with Chart.js graphs, a menu CMS with image uploads, inventory management, one-click refunds via the Stripe API, CSV export, and thermal receipt printing — all behind JWT authentication.

Transactional Email & SMS

Branded HTML email templates via Resend for order confirmations, fulfillment notifications, and low stock alerts. Twilio SMS for real-time order status updates so customers don't have to check their inbox.

Scheduled Cron Jobs

Three automated jobs on a schedule: a daily summary email, a weekly revenue report, and an unfulfilled order reminder — keeping the business informed without any manual effort.

Storefront & Cart

Live fuzzy search with keyboard navigation powered by Fuse.js. Shopping cart persisted to localStorage so items survive page refreshes. Customer order notes field so special requests reach the kitchen.

Security & Rate Limiting

JWT authentication on every admin API route. API rate limiting via express-rate-limit. No secrets exposed to the browser — all third-party API keys server-side only.

Product Thinking

  • Scheduled pickup time slots with slot disabling. Past times are disabled, same-day slots within 2 hours are removed, and orders placed after 6pm roll to next-morning slots. A customer placing an order at 8pm shouldn't be able to select 9pm.
  • Minimum order enforcement before Stripe is called. The checkout button stays disabled until the cart meets the minimum. No partial charge attempts, no error messages from Stripe — the constraint is enforced in the UI before the payment flow starts.
  • Thermal receipt formatting (80mm). The admin's print receipt is formatted for a standard 80mm thermal receipt printer, not a browser print dialog. That's the printer sitting next to every point-of-sale terminal — getting this right makes the dashboard usable in a real kitchen.
  • Local timezone date filtering. An early version filtered orders by UTC date. A 10pm order in Detroit showed up under the wrong day because UTC midnight had already passed. Fixed by converting all date comparisons to the server's local timezone.
  • Refund button with graceful error handling. If the Stripe refund API call fails, the owner sees a human-readable message directing them to the Stripe dashboard. Not a raw error. Not a hanging spinner. A clear next step they can actually follow.

Challenges

Stripe Payment Flow — Payment-First, Then Save

Tips are optional and percentage-based, delivery fees vary by zone, and orders below a minimum need to be blocked before Stripe is ever called. All of this is calculated server-side — trusting the client for cart totals is a security problem.

The key decision was payment-first: the Stripe charge must succeed before the order is written to the database. The frontend creates a PaymentIntent on the server, completes the Stripe flow, and only then does the webhook confirm and save the order. Stripe webhooks retry delivery until the order is recorded, so a server crash between payment and save doesn't create phantom orders.

Stripe Webhook — server.js
app.post('/webhook', express.raw({ type: 'application/json' }), async (req, res) => {
    const sig = req.headers['stripe-signature'];
    let event;

    try {
        event = stripe.webhooks.constructEvent(req.body, sig, process.env.STRIPE_WEBHOOK_SECRET);
    } catch (err) {
        return res.status(400).send(`Webhook Error: ${err.message}`);
    }

    if (event.type === 'payment_intent.succeeded') {
        const paymentIntent = event.data.object;
        const orderId = paymentIntent.metadata.order_id;

        await supabase
            .from('orders')
            .update({ status: 'paid', stripe_payment_id: paymentIntent.id })
            .eq('id', orderId);

        await sendOrderConfirmationEmail(orderId);
    }

    res.json({ received: true });
});

JWT Authentication on Admin Routes

The admin dashboard can issue refunds, delete menu items, and export customer data. Every API route behind it needed to be protected so that only authenticated users could reach it — hiding a button in the browser isn't enough.

Supabase Auth handles login and issues a JWT. On every admin API request, a middleware function verifies the token server-side before any data is read or written.

Auth Middleware — middleware/auth.js
const requireAuth = async (req, res, next) => {
    const authHeader = req.headers.authorization;
    if (!authHeader?.startsWith('Bearer ')) {
        return res.status(401).json({ error: 'No token provided' });
    }

    const token = authHeader.split(' ')[1];
    const { data: { user }, error } = await supabase.auth.getUser(token);

    if (error || !user) {
        return res.status(401).json({ error: 'Invalid or expired token' });
    }

    req.user = user;
    next();
};

// Applied to every admin route
app.use('/api/admin', requireAuth);

Scheduled Cron Jobs

The admin needed to receive a daily summary, a weekly revenue report, and alerts for unfulfilled orders — without having to remember to check anything manually.

Three cron jobs run server-side using node-cron. Each queries Supabase for the relevant data, formats it, and sends via Resend. The unfulfilled order reminder only fires if there are actually open orders — no noise when the queue is clear.

Cron Jobs — jobs/scheduler.js
import cron from 'node-cron';
import { sendDailySummary, sendWeeklyReport, sendUnfulfilledReminder } from './emailJobs.js';

// Daily summary — 8am every day
cron.schedule('0 8 * * *', sendDailySummary);

// Weekly report — Monday 9am
cron.schedule('0 9 * * 1', sendWeeklyReport);

// Unfulfilled order reminder — every 4 hours
cron.schedule('0 */4 * * *', sendUnfulfilledReminder);

Refund Button — Idempotency and Graceful Failure

The dashboard needed a refund button so the business owner could process returns without logging into Stripe. Two problems: pressing the button twice shouldn't issue two refunds, and when Stripe does fail, the owner needs a clear next step — not a raw error message.

The refund endpoint checks whether the order already has a refund_id in Supabase before calling Stripe. If it does, it returns the existing refund. If the Stripe call fails, the UI shows a human-readable message directing the owner to the Stripe dashboard — a place they can actually go to resolve it.
